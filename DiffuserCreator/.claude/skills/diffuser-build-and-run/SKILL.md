# DiffuserCreator Build & Environment

How to work on this project without breaking it. There is no CI and no test project — verification is manual in the editor. Verified 2026-07-06 against `master`.

## Versions & dependencies

- **Unity 2022.1.15f1** (exact; `ProjectSettings/ProjectVersion.txt`). This is an old, non-LTS editor — expect to be prompted to upgrade on first open; upgrading is a deliberate decision, not automatic.
- Key packages (`Packages/manifest.json`): `com.unity.probuilder` 6.0.8, `com.unity.formats.fbx` 5.1.4, `com.unity.timeline`, `com.unity.ugui` + `com.unity.ui.builder` (UI Toolkit — `Assets/MainMenu.uxml`), TextMesh Pro (`Assets/TextMesh Pro`).
- **ProBuilder is a real code dependency**, not just a tool: `DiffuserBlock.SetDepthWithAngleCurve` calls `UnityEngine.ProBuilder.Snapping.Snap`. Removing ProBuilder breaks compilation.
- **TextMesh Pro** is a code dependency via `VertexIndicator` (`TMPro.TextMeshPro`).

## Trap 1 — editor-only APIs in runtime scripts

`Assets/Scripts/` has **no asmdef**, so every script compiles into `Assembly-CSharp` (a runtime assembly). But two scripts call `UnityEditor`:
- `ObjExporterScript.cs` — `using UnityEditor;`, `[MenuItem]`, `EditorUtility.SaveFilePanel`, `Selection`.
- `DiffuserGrid.SaveMesh` / `SaveAsMesh` — `EditorUtility`, `FileUtil`, `AssetDatabase`, `MeshUtility`.

In the editor this compiles fine. **In a player build it will not compile** (`UnityEditor` is stripped). The whole thing works today only because nobody has made a standalone build. When touching this code, either:
1. wrap the editor-only members in `#if UNITY_EDITOR … #endif`, or
2. move `ObjExporter`/`SaveMesh` into an `Editor/` folder with an Editor-only asmdef (the clean fix), keeping runtime scripts editor-free.

Prefer option 2 if you're already reorganizing; option 1 is the minimal safe patch.

## Trap 2 — renaming serialized fields loses data

`DiffuserGrid` and `DiffuserBlock` have their configuration set in `DiffuserGrid.prefab`, `DiffuserBlock.prefab`, and `MainScene.unity`. Unity serializes by **field name**. Renaming any `public` or `[SerializeField]` field (`_rows`, `_columns`, `_blockPrefab`, `SelectedCurveMode`, `SnapAngle`, `UseHorizontalCurve`, `DioganalCurve`, `_cuttingLayerMask`, the material fields on `SelectableBlock`, etc.) **silently resets the stored value to default** on next import. Since you can't open the editor from here to re-enter values, always preserve the old name:

```csharp
using UnityEngine.Serialization;

[FormerlySerializedAs("DioganalCurve")]
public AnimationCurve DiagonalCurve;
```

This is the correct way to fix the pervasive **"Dioganal"** misspelling (fields `DioganalCurve`, `UseDioganalCurve`) — rename the C# member for readability, keep the serialized data with `[FormerlySerializedAs]`. Private non-serialized fields (`_dioganalCurve` inside `DiffuserBlock`) rename freely.

Changing a field's **type**, or moving a script to a different **namespace/class name**, likewise breaks the `MonoBehaviour` GUID/type reference in prefabs — adding a namespace to these `MonoBehaviour`s is safe only because Unity keys components by the script's file GUID (in the `.cs.meta`), not the type name; the class name itself must stay the same. Do not rename the class or the `.cs` file.

## PDFsharp — vendored but not wired in

`PDFsharp/` at the repo root is a full separate solution (`PdfSharp.sln`), not under `Assets/`, so Unity does not compile it. No `Assets/Scripts` file references it. It was presumably added toward a "print the diffuser as a PDF blueprint" feature that was never connected. Treat it as dead weight unless/until a PDF export feature is actually built; don't assume calling it "just works" inside Unity (it targets a different runtime/csproj).

## The `.csproj`/`.sln` at the root are generated

`Assembly-CSharp.csproj`, `RuntimeTransformHandle.csproj`, `DiffuserCreator.sln`, etc. are **regenerated by Unity/Rider** and should not be hand-edited or relied upon as source of truth — they show up as modified in git after any editor open. `.DotSettings` files hold the ReSharper/Rider formatting config that matches `.editorconfig`.

## Verifying a change (no CI)

1. Open the project in Unity 2022.1.15f1; let it recompile. Watch the Console for compile errors (the fastest signal).
2. Open `MainScene.unity`, press Play — `DiffuserGrid.Start()` regenerates the grid; a healthy wall of blocks appears.
3. Exercise the `[ContextMenu]` actions on `DiffuserGrid` and check the Console.
4. For serialized-field changes, confirm the prefab/scene still shows your configured values (not reset to 0/empty) after recompile — this catches a missing `[FormerlySerializedAs]`.
5. `git status` will list regenerated `.csproj`/settings + Unity `Library` churn; don't commit generated project files or `Library/`, `Temp/`, `Logs/`, `obj/`, `bin/`.

## Third-party code — do not refactor

`Assets/Plugins/RuntimeTransformHandle/` is a vendored plugin (namespace `RuntimeHandle`). Leave it alone; only `SelectionManager` consumes it.
